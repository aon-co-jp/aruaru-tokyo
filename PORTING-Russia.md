# PORTING.md — Переносимые файлы

Список шаблонов реализации, которые можно перенести в другие проекты как есть (или с незначительными изменениями).

## `markdown_to_rs()` (src/main.rs)

Функция, преобразующая каждую строку Markdown в стиль rustdoc-комментария с префиксом `//!`. Универсально применима к README.md/CLAUDE.md/PORTING.md. Может быть скопирована как есть в другие репозитории, принявшие концепцию readme-to-rs.

```rust
fn markdown_to_rs(markdown: &str) -> String {
    markdown
        .lines()
        .map(|line| if line.is_empty() { "//!".to_string() } else { format!("//! {line}") })
        .collect::<Vec<_>>()
        .join("\n")
        + "\n"
}
```

## `fetch_repo_file()` (src/main.rs)

Вспомогательная функция, получающая необработанный контент GitHub без аутентификации (с резервным переходом main→master). Может быть повторно использована для других функций интеграции с GitHub, просто заменив имя организации и значение тайм-аута.

## Паттерн nginx «приоритет conf.d»

`/etc/nginx/conf.d/*.conf` подключается в `nginx.conf` раньше, чем `/etc/nginx/sites-enabled/*.conf`, поэтому размещение конфигурации с тем же `server_name` в `conf.d/` даёт ей приоритет над конфигурациями, автоматически генерируемыми UI-инструментами (например, aruaru-easyweb) в `sites-enabled/`. Переиспользуемая техника при возникновении подобного конфликта на другом домене (подробности см. в `CLAUDE.md`).

## API GitHub: различие между эндпоинтами для Organization и личного аккаунта

`GET /orgs/{name}/repos` работает только для GitHub Organization; вызов для личного аккаунта (User) возвращает 404 (подтверждено на реальной инфраструктуре). Чтобы получить полный список репозиториев личного аккаунта, используйте вместо этого `GET /users/{name}/repos`. `aon-co-jp` — личный аккаунт, а не Organization.

```rust
// ✗ Не работает для личного аккаунта (возвращает 404)
let url = format!("https://api.github.com/orgs/{name}/repos?per_page=100");
// ✓ Правильно для личного аккаунта
let url = format!("https://api.github.com/users/{name}/repos?per_page=100");
```

При переносе функций интеграции с GitHub в другие проекты либо заранее проверьте через `GET /users/{name}` (является ли поле `"type"` значением `"Organization"` или `"User"`), либо рассмотрите резервный вариант, пробующий оба эндпоинта.

## `markdown_to_github_style_html()` + `pulldown-cmark` (src/main.rs)

Функция, которая фактически рендерит внешний README в стиле GitHub для отображения. Шаблон для сочетания этого с представлением преобразования `.rs` (`markdown_to_rs`) и переключения между ними через вкладки.

```rust
fn markdown_to_github_style_html(markdown: &str) -> String {
    use pulldown_cmark::{html, Options, Parser};
    let mut options = Options::empty();
    options.insert(Options::ENABLE_TABLES);
    options.insert(Options::ENABLE_STRIKETHROUGH);
    options.insert(Options::ENABLE_TASKLISTS);
    let parser = Parser::new_ext(markdown, options);
    let mut html_out = String::new();
    html::push_html(&mut html_out, parser);
    html_out
}
```

Подготовка CSS в стиле `.markdown-body` в месте отображения (подчёркивание заголовков, блоки кода, границы таблиц и т. д.) приближает внешний вид к GitHub.

## Выход за пределы узкого контейнера `main` (центрирование на основе vw)

Когда вся страница должна оставаться узкой для читаемости (например, `max-width: 780px`), но определённый раздел (например, отображение README) должен растягиваться на всю ширину экрана:

```css
section.wide {
  width: 94vw;
  max-width: 1400px;
  position: relative;
  left: 50%;
  transform: translateX(-50%);
}
```

Это создаёт широкий раздел, центрированный относительно окна просмотра, независимо от `max-width` родительского контейнера.

## Проверка динамически полученных списков форматной проверкой (без сопоставления с белым списком)

Когда нужно принимать как жёстко закодированный при запуске сервера список (например, `GITHUB_REPOS`), так и список, полученный динамически из API во время выполнения, проверка ввода через «сопоставление со статическим списком» отклонит новые значения, существующие только в динамическом списке. Вместо этого проверяйте через «правильность формата» (только буквенно-цифровые символы/дефис/подчёркивание/точка, ограничения длины и т. д.).

## Паттерн «зеркальная location» (внутреннее проксирование через переписывание заголовка Host)

Когда нужно внутренне проксировать только определённые пути под одним доменом к реальному контенту другого домена:

```nginx
location /some-path/ {
    proxy_pass http://127.0.0.1:80/some-path/;
    proxy_set_header Host other-domain.example;
    proxy_set_header X-Real-IP $remote_addr;
}
```

Переиспользуемо всякий раз, когда на одном VPS сосуществуют несколько доменов и нужно, чтобы контент был доступен для просмотра с другого домена без дублирования.
