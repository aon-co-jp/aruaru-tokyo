### 1. Справочные материалы по более новым языкам/фреймворкам

Сам Rust имеет долгую историю, но относительно новые веб-фреймворки с малым объёмом информации, принятые в этой экосистеме, такие как [Poem](https://github.com/poem-web/poem), имеют гораздо меньше обучающих данных, опубликованных примеров, вопросов-ответов и освещения в блогах, чем широко распространённые комбинации вроде Python+FastAPI. Из-за этого при разработке с помощью ИИ (Claude и т. д.) часто возникают ошибки в запоминании API, использование устаревших версий API и другие повторяющиеся паттерны сбоев, которые уже неоднократно случались в этом проекте.

В качестве контрмеры, прежде чем приступать к задаче, ИИ следует сначала посмотреть только ту часть таблицы ниже, которая относится к задаче (нет нужды читать всё — достаточно бегло просмотреть 1–2 релевантные записи). Это должно повысить эффективность и снизить количество переделок, связанных с разработкой на основе ИИ.

### 2. Размышления об инструментах разработки на основе ИИ (2026-07-15, зафиксировано как наблюдение пользователя)

По состоянию на 2026-07-15 универсальные чат-инструменты ИИ, такие как ChatGPT, могут разрабатывать небольшие веб-приложения, но объём переделок значительно растёт, как только система становится умеренно сложной/крупной, и существует жёсткий предел размера программы, с которым можно работать одновременно.

## Роль этого репозитория

Главная страница `aruaru.tokyo`. 2026-07-15 реализация, ранее написанная на PHP, была переписана на Rust + [Poem](https://github.com/poem-web/poem) (по указанию пользователя: «aruaru.tokyo должен быть основан на Rust+Poem»). `audiocafe.tokyo` остаётся на PHP — намеренное решение, при котором стек различается в зависимости от домена.

## Технологический стек

- Rust + Poem (лёгкий веб-фреймворк на основе hyper). Без зависимости от БД, единый бинарный файл.
- Без тяжёлых фреймворков/ORM (согласно соглашению экосистемы poem-cosmo-tauri).
- Фронтенд — это обычный HTML, собираемый на сервере в виде строк (без движка шаблонов). JS ограничен встроенным в страницу `<script>` (поведение кнопки shuffle).

## Основные модули (единый файл `src/main.rs`)

- `categories()` — статические данные для контента «аруару» (повседневные узнаваемые моменты)
- `render_related_sites()` / `RELATED_SITES` — ссылки на связанные страницы на audiocafe.tokyo
- `fetch_repo_file()` / `markdown_to_rs()` — получает необработанный контент GitHub и преобразует его в текст в стиле `.rs` (функция readme-to-rs)
- `markdown_to_github_style_html()` — использует `pulldown-cmark` для фактического рендеринга README и т. д. в HTML в стиле GitHub (доступно через переключаемую вкладку рядом с преобразованием `.rs`)
- `fetch_org_repos()` / обработчик `api_repos` (`GET /api/repos`) — каждый раз получает актуальный полный список имён репозиториев `aon-co-jp` из API GitHub. **Примечание: `aon-co-jp` — личный аккаунт, а не Organization, поэтому нужно использовать эндпоинт `/users/{name}/repos`** (подтверждено на реальной инфраструктуре, что `/orgs/` возвращает 404). Подвержен ограничению скорости API GitHub (60/час/IP), так как вызовы неаутентифицированы.
- `is_valid_repo_name()` — проверяет формат имени репозитория. Поскольку динамически полученные репозитории отсутствуют в статическом списке `GITHUB_REPOS`, для их принятия используется эта проверка формата (а не сопоставление со статическим списком).
- Обработчик `top()` — собирает весь HTML главной страницы (принимает параметр `?repo=` через `Query<TopQuery>`)

## Развёртывание

Запускает `cargo build --release` напрямую на VPS (ConoHa, AlmaLinux), привязывая полученный бинарный файл как службу systemd (`aruaru-tokyo-server.service`) на `127.0.0.1:4100`. nginx (`/etc/nginx/conf.d/aruaru.tokyo.conf`) завершает TLS на порту 443 и выполняет обратное проксирование через `location /` на этот порт.

**Зеркальные location `/aruaru/`, `/aruaru-lady/`, `/rakuten-mobile/`**: эти пути изначально существуют только на стороне `audiocafe.tokyo` (PHP); чтобы сделать их доступными для просмотра напрямую и с `aruaru.tokyo`, отдельные блоки `location` nginx внутренне проксируют на `http://127.0.0.1:80/` с переписанным заголовком Host на `audiocafe.tokyo`. Сам этот бинарный файл никогда не обрабатывает эти пути (в `Route` зарегистрированы только `/` и `/healthz`).

## Правила эксплуатации

- Не добавлять в этот репозиторий функции, зависящие от БД, или тяжёлые фреймворки.
- Всегда использовать публичный URL для констант внешних ссылок (loopback-адрес недостижим для браузеров посетителей).
- Никогда не фиксировать реальный IP-адрес VPS в коде или документации.
- При каждом обновлении CLAUDE.md также обновлять эти версии на 10 языках (`CLAUDE-<Язык>.md`) с тем же содержимым и делать push вместе.

## Роль уровня сервера приложений (open-runo / poem-cosmo-tauri, добавлено 2026-07-16)

`open-web-server` был добавлен как вариант для «движка доставки (vhost)», но пока `open-web-server` ещё не функционирует как полноценный гибридный веб-сервер Apache+Nginx, роль совместимого слоя в стиле Tomcat выполняют `open-runo` или `poem-cosmo-tauri`.

Они, вместе с `open-raid-z` и VersionlessAPI, нацелены на функционирование как фреймворк/промежуточное ПО, полностью обеспечивающее критически важную для бизнеса, круглосуточную и бесперебойную веб-разработку 24/7/365 — для случаев использования, где недопустима потеря данных в интернете, таких как покупки предметов в 3D-онлайн-играх, онлайн-финансы, онлайн-биржевая торговля и онлайн-платежи по кредитным картам — путём построения передовой системы связи с «четырёхуровневой, четырёхкратной избыточностью», сочетающей `aruaru-db` (совместимую с ACID и ZFS) с конфигурацией DUAL DATABASE наряду с PostgreSQL, на основе дизайна базы данных, облегчающего изменения спецификаций при сохранении безверсионной работы наряду с контролем версий и управлением Git.
