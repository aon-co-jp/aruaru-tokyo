# PORTING.md — Portable Files

A list of implementation patterns that can be carried over to other projects as-is (or with minor changes).

## `markdown_to_rs()` (src/main.rs)

A function that converts each line of Markdown into `//!`-prefixed rustdoc-comment style. Generically usable for README.md/CLAUDE.md/PORTING.md alike. Can be copied as-is into other repos that adopt the readme-to-rs concept.

```rust
fn markdown_to_rs(markdown: &str) -> String {
    markdown
        .lines()
        .map(|line| if line.is_empty() { "//!".to_string() } else { format!("//! {line}") })
        .collect::<Vec<_>>()
        .join("\n")
        + "\n"
}
```

## `fetch_repo_file()` (src/main.rs)

A helper that fetches GitHub raw content without authentication (with a main→master fallback). Can be repurposed for other GitHub-integration features just by swapping the org name and timeout value.

## nginx "conf.d precedence" pattern

`/etc/nginx/conf.d/*.conf` is included in `nginx.conf` before `/etc/nginx/sites-enabled/*.conf`, so placing a config with the same `server_name` in `conf.d/` lets it take precedence over configs auto-generated by UI tools (e.g. aruaru-easyweb) into `sites-enabled/`. A reusable technique whenever the same conflict arises on another domain (see `CLAUDE.md` for details).

## GitHub API: the difference between Organization endpoints and personal-account endpoints

`GET /orgs/{name}/repos` only works for GitHub Organizations; calling it against a personal account (User) returns 404 (confirmed on real infrastructure). To get a personal account's full repo list, use `GET /users/{name}/repos` instead. `aon-co-jp` is a personal account, not an Organization.

```rust
// ✗ Doesn't work for a personal account (returns 404)
let url = format!("https://api.github.com/orgs/{name}/repos?per_page=100");
// ✓ Correct for a personal account
let url = format!("https://api.github.com/users/{name}/repos?per_page=100");
```

When porting GitHub-integration features to other projects, either check in advance via `GET /users/{name}` (whether the `"type"` field is `"Organization"` or `"User"`), or consider a fallback that tries both endpoints.

## `markdown_to_github_style_html()` + `pulldown-cmark` (src/main.rs)

A feature that actually renders an external README GitHub-style for display. A pattern for pairing it with the `.rs`-conversion view (`markdown_to_rs`) and letting the user toggle between them via tabs.

```rust
fn markdown_to_github_style_html(markdown: &str) -> String {
    use pulldown_cmark::{html, Options, Parser};
    let mut options = Options::empty();
    options.insert(Options::ENABLE_TABLES);
    options.insert(Options::ENABLE_STRIKETHROUGH);
    options.insert(Options::ENABLE_TASKLISTS);
    let parser = Parser::new_ext(markdown, options);
    let mut html_out = String::new();
    html::push_html(&mut html_out, parser);
    html_out
}
```

Preparing `.markdown-body`-style CSS at the display destination (heading underlines, code blocks, table borders, etc.) brings it closer to GitHub's actual look.

## Breaking out of a narrow `main` container (vw-based centering)

When you want the overall page to stay narrow (e.g. `max-width: 780px`) for readability, but need one specific section (like a README display) to expand to the full screen width:

```css
section.wide {
  width: 94vw;
  max-width: 1400px;
  position: relative;
  left: 50%;
  transform: translateX(-50%);
}
```

This creates a wide, viewport-centered section regardless of the parent container's `max-width`.

## Validate dynamically-fetched lists by format check (not whitelist matching)

When you want to accept both a hardcoded list at server startup (e.g. `GITHUB_REPOS`) and a list fetched dynamically from an API at runtime, validating input by "matching against the static list" will reject new values that only exist in the dynamic list. Instead, validate by "is the format correct" (alphanumeric/hyphen/underscore/dot only, length limits, etc.).

## The "mirror location" pattern (internal proxying via Host-header rewriting)

When you want to internally proxy only specific paths under one domain to the real content on another domain:

```nginx
location /some-path/ {
    proxy_pass http://127.0.0.1:80/some-path/;
    proxy_set_header Host other-domain.example;
    proxy_set_header X-Real-IP $remote_addr;
}
```

Reusable whenever multiple domains coexist on the same VPS and you want content viewable from another domain without duplicating it.
